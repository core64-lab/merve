# ML Server Deployment Strategy

## Overview
This document outlines the deployment strategy for ML classifiers using the merve framework, supporting both single and multi-classifier repositories with proper versioning.

## Key Design Decisions

### 1. API Versioning Strategy
We recommend **removing the API version from URLs** and using **versionless endpoints** with metadata responses. Here's why:

- **Immutable containers**: Each Docker image is a specific version of a classifier
- **Version in metadata**: Every response includes repository, classifier name, and git tag
- **Kubernetes manages versions**: Different versions run as separate deployments
- **Simpler client integration**: Clients don't need to update URLs for version changes

### 2. Multi-Classifier Repository Support
A single repository can contain multiple classifiers, each with its own:
- Predictor class
- Model artifacts
- Configuration file
- Docker image

## Repository Structure

### Single Classifier Repository
```
my-classifier/
├── mlserver.yaml           # Main configuration
├── predictor.py           # Predictor class
├── artifacts/             # Model files
│   ├── model.pkl
│   └── preprocessor.pkl
└── Dockerfile            # Auto-generated
```

### Multi-Classifier Repository
```
multi-classifier-repo/
├── mlserver.yaml          # ONE multi-classifier config (top-level `classifiers:` section)
├── predictor_sentiment.py # Predictor class per classifier
├── predictor_intent.py
├── predictor_ner.py
├── artifacts/             # Model files per classifier
│   ├── sentiment/
│   ├── intent/
│   └── ner/
└── shared/                # Shared utilities
    └── preprocessing.py
```

One `merve build` produces a single commit image bundling all classifiers (see the build strategy below); each deployment picks its classifier via `MLSERVER_CLASSIFIER`.

## Versionless Endpoints and Metadata

Endpoints are versionless and flat: every deployment serves exactly one classifier at `/predict`, `/predict_proba`, `/info`, `/status`, `/healthz`, and `/metrics` — no version or classifier name in the URL. Version information travels in the response metadata instead:

```json
{
  "predictions": [...],
  "time_ms": 12.5,
  "predictor_class": "SentimentPredictor",
  "metadata": {
    "project": "multi-classifier-repo",
    "classifier": "sentiment-analyzer",
    "predictor_class": "SentimentPredictor",
    "predictor_module": "predictor_sentiment",
    "config_file": "mlserver.yaml",
    "git_commit": "abc123def",
    "git_tag": "sentiment/v2.3.1",
    "deployed_at": "2024-01-15T10:30:00Z",
    "mlserver_version": "0.3.2",
    "mlserver_api_commit": "b5dff2a",
    "mlserver_api_tag": null
  }
}
```

## CI/CD Workflow Adaptations

### 1. Git Tagging Strategy

Git tags are the canonical version source for each classifier.

#### Tag Format
```
Canonical (created by `merve tag`): <classifier>/vX.Y.Z                  (e.g. sentiment/v2.3.1)
Legacy (still read for old tags):      <classifier>-vX.Y.Z-mlserver-<hash>  (e.g. sentiment-v2.3.1-mlserver-b5dff2a)
```

`merve tag` creates the **canonical** `<classifier>/vX.Y.Z` form. The MLServer commit is no longer part of the tag name — it is recorded in the annotated-tag message and in the container's OCI image labels. The **legacy** form is still **read** everywhere (status, build validation, version listing), so existing tags keep working.

> **CI trigger:** the GitHub Actions workflow generated by `merve init-github` (workflow version 3) triggers on the canonical pattern `'*/v*'` and parses `<classifier>/vX.Y.Z` tags. Only hand-written workflows that predate the canonical format still need their trigger updated from the legacy `'*-v*-mlserver-*'` pattern — see the migration section below.

#### Creating Tags

Use the CLI tool (the MLServer commit is recorded in the annotated-tag message and OCI labels):
```bash
# Create tags using merve CLI
merve tag --classifier sentiment patch   # v2.3.1
merve tag --classifier intent minor      # v1.1.0
merve tag --classifier ner major         # v3.0.0

# Tags created: sentiment/v2.3.1
#                intent/v1.1.0
#                ner/v3.0.0

# Push tags to remote
git push --tags
```

#### View Tag Status

### 2. Build Strategy for Multi-Classifier (build-once / deploy-many)

For multi-classifier repositories the **default** is one image per git commit:

```bash
# ONE commit image bundling every classifier - no per-classifier build loop
merve build
# → <repo>:<git-sha7> and <repo>:latest (no baked classifier)

# The release is applied as registry tag aliases on that same image (no rebuild):
merve push --classifier sentiment --registry registry.example.com
# → <repo>:sentiment-v2.3.1 and <repo>:sentiment-latest (validated against the git tag)
```

The classifier is selected at deploy/run time via the `MLSERVER_CLASSIFIER`
environment variable (`merve run --classifier sentiment` sets it for you).
`merve build --per-classifier-image --classifier <name>` is the escape hatch for
classifiers whose conflicting dependencies cannot share one image.

### 3. GitHub Actions Workflows

`merve init-github` generates a complete workflow (version 3) implementing
exactly this pattern; the examples below show condensed hand-written variants.

#### Option A: Trigger on Git Tag Push (Recommended)

`.github/workflows/build-on-tag.yml`:
```yaml
name: Build and Deploy on Tag

on:
  push:
    tags:
      - '*/v*'  # canonical release tag <classifier>/vX.Y.Z

jobs:
  parse-and-build:
    runs-on: ubuntu-latest
    steps:
      - name: Parse Canonical Tag
        id: parse
        run: |
          TAG_NAME="${GITHUB_REF#refs/tags/}"
          echo "Full tag: $TAG_NAME"

          # Parse: sentiment/v2.3.1
          CLASSIFIER="${TAG_NAME%%/v*}"
          VERSION="${TAG_NAME##*/v}"
          COMMIT_SHA="${GITHUB_SHA:0:7}"

          if [ "$CLASSIFIER" = "$TAG_NAME" ] || [ -z "$VERSION" ]; then
            echo "❌ Tag '$TAG_NAME' is not in canonical <classifier>/vX.Y.Z form"
            exit 1
          fi

          echo "classifier=$CLASSIFIER" >> $GITHUB_OUTPUT
          echo "version=$VERSION" >> $GITHUB_OUTPUT
          echo "commit_sha=$COMMIT_SHA" >> $GITHUB_OUTPUT

      - uses: actions/checkout@v4
        with:
          ref: ${{ github.ref }}
          fetch-depth: 0  # Fetch all history for git operations
          fetch-tags: true

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install Merve
        run: |
          pip install merve

      - name: Build Commit Image
        run: |
          # Build ONCE per commit - the image bundles every classifier
          merve build

      - name: Login to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ secrets.REGISTRY_URL }}
          username: ${{ secrets.REGISTRY_USERNAME }}
          password: ${{ secrets.REGISTRY_PASSWORD }}

      - name: Push Commit Image and Release Alias
        run: |
          CLASSIFIER="${{ steps.parse.outputs.classifier }}"
          VERSION="${{ steps.parse.outputs.version }}"
          COMMIT_SHA="${{ steps.parse.outputs.commit_sha }}"
          REPO="my-repo"  # repository image name from mlserver.yaml

          # One commit image per git commit, shared by every classifier
          docker tag ${REPO}:latest ${{ secrets.REGISTRY_URL }}/${REPO}:${COMMIT_SHA}
          docker push ${{ secrets.REGISTRY_URL }}/${REPO}:${COMMIT_SHA}

          # Per-classifier release alias on the SAME image digest (no rebuild)
          docker tag ${REPO}:latest ${{ secrets.REGISTRY_URL }}/${REPO}:${CLASSIFIER}-v${VERSION}
          docker tag ${REPO}:latest ${{ secrets.REGISTRY_URL }}/${REPO}:${CLASSIFIER}-latest
          docker push ${{ secrets.REGISTRY_URL }}/${REPO}:${CLASSIFIER}-v${VERSION}
          docker push ${{ secrets.REGISTRY_URL }}/${REPO}:${CLASSIFIER}-latest

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          name: "${{ steps.parse.outputs.classifier }} v${{ steps.parse.outputs.version }}"
          body: |
            ## ${{ steps.parse.outputs.classifier }} v${{ steps.parse.outputs.version }}

            **Classifier Commit**: ${{ github.sha }}
            **Container**: `${{ secrets.REGISTRY_URL }}/my-repo:${{ steps.parse.outputs.classifier }}-v${{ steps.parse.outputs.version }}`
            (run with `MLSERVER_CLASSIFIER=${{ steps.parse.outputs.classifier }}`)

            ### Reproducibility
            ```bash
            git checkout ${{ github.ref_name }}
            merve build
            ```
```

#### Option B: Manual Workflow Dispatch

`.github/workflows/build-dispatch.yml`:
```yaml
name: Manual Build and Deploy

on:
  workflow_dispatch:
    inputs:
      full_tag:
        description: 'Canonical release tag (e.g., sentiment/v2.3.1)'
        required: true
        type: string

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Validate Tag Format
        run: |
          TAG="${{ github.event.inputs.full_tag }}"
          if ! echo "$TAG" | grep -qE '^[a-z0-9][a-z0-9_-]*/v[0-9]+\.[0-9]+\.[0-9]+$'; then
            echo "❌ Invalid tag format"
            echo "Expected canonical form: <classifier>/vX.Y.Z"
            echo "Got: $TAG"
            exit 1
          fi
          echo "✅ Tag format valid"

      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.inputs.full_tag }}

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install and Build
        run: |
          pip install merve
          # --classifier with a full tag validates the checked-out code
          # against the tag (--force skips the interactive prompt in CI)
          merve build --classifier ${{ github.event.inputs.full_tag }} --force
```

### 4. Kubernetes Deployment
Each classifier version gets its own deployment. With build-once/deploy-many the
image is the shared commit image (addressed via its release alias) and the
classifier is selected at deploy time with the `MLSERVER_CLASSIFIER` environment
variable — this is fully supported: `merve serve` (the container entrypoint)
reads `MLSERVER_CLASSIFIER`, and an invalid value fails loudly at startup
listing the available classifiers. In Kubernetes that means a typo produces a
crash-looping pod with a clear error instead of silently serving the wrong
model — the desired fail-fast behavior.

`k8s/classifier-deployment.yaml`:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sentiment-analyzer-v2-3-1
  labels:
    app: sentiment-analyzer
    version: v2.3.1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: sentiment-analyzer
      version: v2.3.1
  template:
    metadata:
      labels:
        app: sentiment-analyzer
        version: v2.3.1
    spec:
      containers:
      - name: classifier
        # Release alias on the commit image (bundles all classifiers)
        image: registry.example.com/multi-classifier-repo:sentiment-v2.3.1
        env:
        - name: MLSERVER_CLASSIFIER  # deploy-time classifier selection
          value: "sentiment"
        # Concurrency is configured in mlserver.yaml (api.max_concurrent_predictions,
        # default 1 = one prediction per pod; overflow gets 503 so the LB retries)
        resources:
          requests:
            memory: "512Mi"
            cpu: "500m"
          limits:
            memory: "1Gi"
            cpu: "1000m"
---
apiVersion: v1
kind: Service
metadata:
  name: sentiment-analyzer
spec:
  selector:
    app: sentiment-analyzer
    version: v2.3.1  # Pin to specific version
  ports:
  - port: 8000
    targetPort: 8000
```

### 5. Service Mesh for Canary Deployments
Use Istio or similar for traffic splitting:
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: sentiment-analyzer
spec:
  http:
  - match:
    - uri:
        prefix: /sentiment-analyzer
    rewrite:
      uri: /   # MLServer serves flat endpoints (/predict), so strip the routing prefix
    route:
    - destination:
        host: sentiment-analyzer
        subset: v2-3-1
      weight: 90
    - destination:
        host: sentiment-analyzer
        subset: v2-4-0
      weight: 10  # Canary deployment
```

## Development Workflow

### 1. Local Development
```bash
# Work on a specific classifier (multi-classifier config at the repo root)
merve serve --classifier sentiment

# Test endpoint (flat path - no classifier name in the URL)
# Send input keys at the top level (the legacy {"payload": {...}} wrapper is deprecated)
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{"records": [{"text": "Great product!"}]}'
```

### 2. Releasing Multiple Classifiers (build once)
```bash
# Tag each classifier release independently
merve tag --classifier sentiment patch
merve tag --classifier intent minor

# Look up a classifier's latest canonical release tag if you need it
latest_tag=$(git describe --tags --match "sentiment/v*" --abbrev=0)

# Build ONCE per commit - the commit image bundles every classifier
merve build

# Apply each release as registry tag aliases on that same image (no rebuild)
merve push --classifier sentiment --registry registry.example.com
merve push --classifier intent --registry registry.example.com
```

### 3. Version Bumping

**Recommended**: Use the `merve tag` command for automatic semantic-version tagging:

```bash
# Patch release (bug fixes) - auto-increments version; MLServer commit recorded in tag message + OCI labels
merve tag --classifier sentiment patch

# Output:
# ✓ Created tag: sentiment/v1.0.3
#
#   📝 Version: 1.0.2 → 1.0.3 (patch bump)
#   🔧 MLServer commit: b5dff2a
#   📦 Classifier commit: c5f9997

# Minor release (new features)
merve tag --classifier intent minor

# Major release (breaking changes)
merve tag --classifier fraud-detection major

# Push tags to remote
git push --tags
```

**Manual Alternative** (not recommended):

If you need custom scripting, create `scripts/bump_version.sh`:
```bash
#!/bin/bash
CLASSIFIER=$1
VERSION_TYPE=$2  # major, minor, patch

# Use the merve CLI command instead of manual tag creation
merve tag --classifier $CLASSIFIER $VERSION_TYPE

# Push to remote
git push --tags
```

**Tag Status Overview**:
```bash
# View status of all classifiers in repository
merve tag

# Shows table with current versions, MLServer commits, and recommended actions
```

## Migration Path for Existing Projects

If you have existing projects using older tagging formats, follow this migration guide:

### Step 1: Understand Your Current Tags (5 minutes)
```bash
# List existing tags
git tag -l

# Existing tags might look like:
# v1.0.0
# v1.0.1
# sentiment-v1.0.0
```

### Step 2: Update to Canonical Tags (10 minutes)
```bash
# For single-classifier repos
# Old format: v1.0.0
# New (canonical) format: <classifier>/v1.0.0

# Create first classifier-scoped tag from current version
merve tag --classifier <your-classifier-name> patch

# Example output:
# ✓ Created tag: sentiment/v1.0.1
```

### Step 3: Update CI/CD Workflows (30 minutes)
1. Update GitHub Actions to the canonical tag format (see workflows above), or regenerate with `merve init-github --force`
2. Change tag triggers to `tags: ['*/v*']` (replacing the legacy `'*-v*-mlserver-*'` pattern)
3. Add tag parsing logic to extract classifier and version (`<classifier>/vX.Y.Z`)

### Step 4: Update Container Build Process (15 minutes)
```bash
# Old command:
# docker build -t my-classifier:v1.0.0 .

# New command with canonical tag:
merve build --classifier sentiment/v1.0.1

# Or let MLServer detect the latest tag automatically:
merve build --classifier sentiment
```

### Step 5: Validate Reproducibility (10 minutes)
```bash
# Inspect the tag (MLServer commit is in the annotated-tag message)
git show sentiment/v1.0.1

# Verify you can rebuild from any historical tag
git checkout sentiment/v1.0.1
merve build --classifier sentiment/v1.0.1
```

### Backward Compatibility Notes

- **Existing containers**: Continue to work without changes
- **Old tags**: Remain in git history, no need to delete
- **Gradual migration**: New releases use canonical tags, old ones remain unchanged
- **API endpoints**: No changes required (versionless design recommended)

## Benefits of Per-Classifier Git-Tag Versioning

1. **Complete Reproducibility**: A tag pins the classifier version, and the MLServer commit is captured in the annotated-tag message and the container's OCI labels
   - Example: `sentiment/v2.3.1` names the release; `git show sentiment/v2.3.1` and the image labels recover the exact MLServer commit
   - Can rebuild identical container months or years later
   - No dependency on external registries for version tracking

2. **Multi-Classifier Support**: Single repository can host multiple related classifiers
   - Each classifier has independent versioning
   - Shared utilities and dependencies across classifiers
   - Centralized repository management

3. **Backward Compatible**: Existing single-classifier repos continue to work
   - Gradual migration path
   - No breaking changes to existing deployments
   - Old tags remain valid

4. **CI/CD Integration**: Seamless GitHub Actions workflow
   - Automatic builds on tag push
   - Tag parsing extracts classifier and version from `<classifier>/vX.Y.Z`
   - Validation ensures code matches tag specifications

5. **Clear Deployment Path**: Each version is a separate immutable deployment
   - Kubernetes deployments per version
   - Easy rollback to any previous version
   - Canary and blue-green deployment support

6. **Simplified Client Integration**: Clients use stable URLs, versions tracked via metadata
   - Versionless, flat endpoints: `/predict` on each classifier's service
   - Version information in response metadata
   - No client code changes for version updates

7. **Developer Experience**: Simple CLI commands handle complexity
   - `merve tag --classifier sentiment patch` - one command for version bump
   - Automatic MLServer commit detection
   - Visual status table for all classifiers

8. **Audit Trail**: Complete version history with traceability
   - Git tags provide permanent version markers
   - Container labels include all version metadata
   - Prometheus metrics track version usage

## Example Client Integration

```python
import requests

class MLClient:
    def __init__(self, base_url: str):
        # base_url points at one classifier's service,
        # e.g. http://sentiment-analyzer.ml.svc:8000
        self.base_url = base_url
        self.last_metadata = None

    def predict(self, records: list):
        # Flat, versionless endpoint
        url = f"{self.base_url}/predict"

        # Send input keys at the top level (the legacy "payload" wrapper is deprecated)
        response = requests.post(url, json={"records": records})
        result = response.json()

        # Track version from metadata
        self.last_metadata = result.get("metadata", {})
        print(f"Used {self.last_metadata.get('classifier')} "
              f"(git: {self.last_metadata.get('git_tag')})")

        return result["predictions"]

# Client doesn't need to know about versions
client = MLClient("http://sentiment-analyzer.internal:8000")
predictions = client.predict([{"text": "Great product!"}])
```

## Monitoring and Observability

### Metrics to Track
- Requests per classifier version
- Response times by version
- Error rates by version
- Version adoption rate (for canary deployments)

### Prometheus Metrics

The built-in `mlserver_model_info{model, version}` gauge is set to 1 at startup and identifies the loaded model. Combine it with the per-request metrics (`mlserver_requests_total`, `mlserver_request_duration_seconds`, ...) and Kubernetes labels (`app`, `version`) to break dashboards down by classifier version. See [Observability Features](./observability.md#available-metrics) for the full metric list.

## Security Considerations

1. **Image Scanning**: All classifier images scanned before deployment
2. **Version Pinning**: Production always uses specific versions, never 'latest'
3. **RBAC**: Different teams can manage different classifiers in same repo
4. **Secrets Management**: Model artifacts encrypted, credentials in K8s secrets

## Next Steps for Implementation

### For New Projects

1. **Set up classifier configuration**
   - Create `mlserver.yaml` with classifier metadata
   - Define predictor class and model artifacts

2. **Initialize version control**
   ```bash
   git init
   git add .
   git commit -m "Initial commit"

   # Create first classifier release tag
   merve tag --classifier <your-classifier-name> patch
   ```

3. **Set up CI/CD pipeline**
   - Copy GitHub Actions workflow from examples above
   - Configure container registry credentials
   - Test with first tag push

4. **Deploy to Kubernetes**
   - Use deployment manifests from this guide
   - Configure resource limits based on model requirements
   - Set up monitoring and observability

### For Existing Projects

1. **Migrate to canonical tags** (see Migration Path section above)
2. **Update CI/CD workflows** to handle new tag format
3. **Test reproducibility** with historical builds
4. **Update documentation** for your team

### For Multi-Classifier Repositories

1. **Organize repository structure** (one config, one `classifiers:` section)
   ```
   my-repo/
   ├── mlserver.yaml          # top-level `classifiers:` section names both
   ├── predictor_a.py
   ├── predictor_b.py
   └── artifacts/
       ├── classifier-a/
       └── classifier-b/
   ```

2. **Tag each classifier independently**
   ```bash
   merve tag --classifier classifier-a patch
   merve tag --classifier classifier-b minor
   ```

3. **Build once, publish each release as an alias, deploy separately**
   ```bash
   merve build   # ONE commit image bundling both classifiers
   merve push --classifier classifier-a --registry <url>   # <repo>:classifier-a-v1.0.1
   merve push --classifier classifier-b --registry <url>   # <repo>:classifier-b-v1.1.0
   # Each deployment selects its classifier via MLSERVER_CLASSIFIER
   ```
